# bldg-server

bldg-server is a virtual world platform where **buildings**, **residents**, and
**batteries** interact. It provides a spatial, hierarchical environment where
people and AI agents coexist, communicate, and render data as a navigable
3D space.

There are **two layers** you can work at — pick the one that matches your goal.

## The two layers

### Which should I use?

| If you want to… | Use |
|---|---|
| Turn an existing dataset (folders/JSON/CSV) into a 3D scene | **Visualize Datasets** — declare it once in `visual-language.json` |
| Build a custom interactive app, agent, or live tool | **Protocol** — call the API and write a battery |
| Understand what the high-level layer compiles down to | Read both — the high-level layer emits the low-level commands for you |

The high-level layer is built **on top of** the protocol: the file-system
battery reads your folder and emits the same `/create`, `/relocate`, and
`/connect` commands you could send by hand. Everything you can express
declaratively, you can also do directly.

## Core ideas (shared by both layers)

- **Buildings** are the fundamental spatial unit. They have addresses,
  coordinates, and can contain other buildings via floors — a recursive
  hierarchy.
- **Residents** are the people and agents that inhabit buildings. They move,
  turn, enter/exit, and communicate via SAY actions.
- **Batteries** are external services that attach to buildings and extend them.
  They receive chat messages via webhooks and respond with actions. The
  file-system battery is one such battery.
- **Chat layer** ties it together: messages starting with `/` become commands
  that create buildings, connect roads, and more.

## API base URL

All protocol endpoints are under `/v1/`:

```
https://bldg-server.fly.dev/v1/
```

For live views that stay in sync with a floor, connect to the WebSocket
[Floor Channel](/streaming/overview) instead of polling.
