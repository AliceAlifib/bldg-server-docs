# Visualizing Datasets

The high-level layer lets you render an entire dataset as a navigable 3D scene
**without writing any code or protocol calls**. You arrange your data as a
folder tree and describe how it should look in a single `visual-language.json`
file. The **file-system battery** reads the folder, derives the visualization,
and emits the underlying protocol commands for you.

<Note>
This layer is implemented by the **file-system battery** — a generic battery
that attaches to a building and renders any folder shaped per the dataset spec.
It is the easiest way to get structured data into bldg-server.
</Note>

## The mental model

```
folder/                     ← the dataset (a "world")
├── visual-language.json    ← the one config: how everything looks + connects
├── semantic-attributes.json  ← (optional) the root is itself a record
├── 001_alpha/              ← a container (a group)
│   ├── 010_record-a/       ← a leaf = one data record
│   │   ├── raw-attributes.json
│   │   └── semantic-attributes.json
│   └── 011_record-b/
│       └── ...
└── 002_beta/
    └── ...
```

- **Each leaf folder is one data record** — a building in the scene.
- **Each parent folder is a container** — it groups its children (and may also
  be a record itself).
- **`visual-language.json` at the root** maps your data to visuals: which 3D
  object each entity type uses, how levels are laid out, which attributes become
  colors or nested labels, and which records are connected by roads.

That's the whole contract. Point the battery at the folder and you get a scene.

## What you can express

<CardGroup cols={2}>
  <Card title="Embodiment" icon="shapes">
    Map each entity type to a 3D object (`entity_types`), and turn attribute
    values into colors/states (`attribute_visual_mappings`).
  </Card>
  <Card title="Hierarchy & layout" icon="diagram-project">
    Control per-depth shape and placement with `hierarchy_levels` and named
    layout strategies (e.g. fishbone spine + branches).
  </Card>
  <Card title="Detail panels" icon="tags">
    Render selected attributes (summary, description, category…) as small
    bldgs *inside* each record with `detail_bldgs`.
  </Card>
  <Card title="Roads" icon="road">
    Connect related records with `connections` (spine chains, parent→children,
    cross-pairs).
  </Card>
</CardGroup>

## How it relates to the protocol

The file-system battery is a thin, declarative front-end over the
[protocol](/concepts/buildings). When you run it, it:

1. Walks the folder into an in-memory model.
2. Builds a container building for the dataset.
3. Emits `/create` for every record (mapping your `entity_types` to the
   bldg-server's [visual language](/concepts/visual-language) 3D objects).
4. Emits `/relocate` to lay records out per your layout strategies.
5. Emits `/connect` for your `connections` rules to draw roads.
6. Emits per-record detail bldgs and attribute data.

<Tip>
Anything the high-level layer does, you can also do by hand with the
[chat commands](/chat-layer/commands) and [Building API](/api-reference/buildings/build).
Use the high-level layer for whole datasets; drop to the protocol for bespoke,
interactive behavior.
</Tip>
