# Floors

Floors are the vertical layers inside composite buildings. They provide the surface on which child buildings and residents are placed.

## Addressing

Floors use the `l<n>` notation in addresses:

```
g/b(10,20)/l0     <- floor 0 inside building at (10,20)
g/b(10,20)/l1     <- floor 1 inside the same building
```

When a composite building is created, **floor 0** (`l0`) is the default floor where entities are placed.

## Nesting Depth

Because buildings can contain floors which contain buildings which contain floors, the hierarchy can nest arbitrarily deep:

```
g                          <- depth 0
└── b(10,20)               <- depth 1
    └── l0                 <- (floor)
        └── b(5,3)         <- depth 2
            └── l0         <- (floor)
                └── b(1,1) <- depth 3
```

The `nesting_depth` field on buildings and residents tracks how deep they are in this hierarchy.

## Floor Heights

Each entity type defines its own floor dimensions, stored in the building's `data` JSON field:

| Entity Type | `flr_height` | `flr0_height` |
|-------------|-------------|---------------|
| `ground` | 1.08 | 0.01 |
| `problem` | 1.08 | 1.11 |
| `stage` | 0.9 | 0.022 |
| `green-lot` | 0.9 | 0.022 |
| `blue-lot` | 0.9 | 0.022 |
| `milestone` | 0.63 | 0.0 |
| `sales` | 3.0 | 1.3 |
| `team` | 0.9 | 0.05 |
| `storage` | 0.7 | 0.03 |
| `costs` | 2.57 | 0.067 |
| `inbox` | 2.57 | 0.067 |

These values are used by visual clients (like the Unity WebGL client) to render the 3D environment correctly.

## Navigating Floors

Residents move between floors using:

- **ENTER_BLDG** — enter a building's default floor (l0)
- **ENTER_BLDG_FLR** — enter a specific floor level with target coordinates
- **EXIT_BLDG** — leave the current building, returning to the containing floor
